# 0.1.6 — 2026-06-30

## Fixed

- **CO-OPS date tools (`noaa_marine_get_tide_predictions`, `noaa_marine_get_water_level`, `noaa_marine_get_currents`)** — invalid calendar dates (`20250231`) and reversed ranges (`begin_date` after `end_date`) now reject locally as `invalid_date_range` (`ValidationError`) with date-specific recovery, instead of reaching CO-OPS and surfacing its HTTP 400 as a misleading `station_not_found`. New shared validator `validateCoopsDateRange` (`src/services/coops/date-range.ts`) strictly parses `YYYYMMDD`; each tool keeps its own span limit. ([#7](https://github.com/cyanheads/noaa-marine-mcp-server/issues/7))
- **`noaa_marine_get_conditions`** — a buoy whose realtime file exists but carries no usable sensor values now preserves the service-level `no_sensor_data` reason and recovery; the handler inspects `err.data.reason` before reclassifying, reserving `buoy_not_found` for genuine 404s. ([#6](https://github.com/cyanheads/noaa-marine-mcp-server/issues/6))
- **`noaa_marine_find_stations`** — a `state` filter with the default `source="all"` no longer admits state-less NDBC buoys; NDBC is excluded whenever `state` is set, so a state-scoped search returns CO-OPS stations rather than global buoys. ([#5](https://github.com/cyanheads/noaa-marine-mcp-server/issues/5))
- **NDBC station names** — blank or whitespace-only `name` attributes in the active-stations XML are normalized to an `NDBC <id>` label (uppercased ID) before caching, so buoys no longer sort first with empty names or render blank in the station resource. ([#8](https://github.com/cyanheads/noaa-marine-mcp-server/issues/8))
